# Superconductor User Dogfood

Date: 2026-05-03

This note records a public-safe dogfood pass from the human operator's seat:
the user is watching Superconductor, a running Claude Code/Codex-style terminal
session, repo changes, and workbench documentation at the same time.

No raw screenshots, private thread contents, tokens, request payloads, or live
run IDs are stored here.

## User Job

The user is not asking, "does the protocol exist?"

The user is asking:

```text
Can I look at this running agent workspace and quickly know:
- what it is trying to do;
- whether it is still making progress;
- what changed;
- what evidence it has;
- what I should do next?
```

That is the dogfood frame. The system should be judged by how much thinking the
operator must do to recover state from the UI.

## Observed Path

The visible workflow had these surfaces at once:

- Superconductor project list and selected worktree.
- A running agent terminal session.
- A file/change/check side panel.
- Workbench docs around Windburn cognitive cache.
- Repo state and untracked artifacts.
- A separate expectation that the agent should act like a user, not only like a
  repo reviewer.

Tool evidence from this pass:

```text
Superconductor worktree status:
branch=main
target_branch=main
agent_state=working
files_changed=5
```

Computer Use window readback failed with:

```text
Apple event error -1743
```

This means app-level visual inspection needed to fall back to the supplied
screenshot plus Superconductor MCP state instead of pretending desktop state was
fully observable.

## Findings

### 1. `working` Is Not Enough State

`agent_state=working` is technically true, but it does not answer the user's
question. The operator still has to infer whether the agent is:

- exploring;
- editing;
- blocked;
- waiting for approval;
- verifying;
- drifting into a new objective.

The missing product primitive is a small status card:

```text
Goal:
Current action:
Last verified evidence:
Changed files:
Waiting on:
Safe next action:
```

### 2. Dirty Changes Need Source Buckets

The UI can show an uncommitted-change count, but the user needs buckets:

```text
intentional source changes
generated evidence
browser/test artifacts
operator-provided assets
unknown/unclaimed files
```

Without buckets, a user sees "5 changes" and has to inspect the repo mentally.
That is not a good dogfood experience for a multi-agent control surface.

### 3. Repo Anchor Must Be Visible

The screenshot shows the user operating inside a workbench project while also
discussing Windburn. This is normal for the current workflow, but it creates a
real user-facing ambiguity:

```text
Am I changing the workbench memory repo, the Windburn product repo, or only a
temporary Superconductor worktree?
```

The repo anchor should be visible in every closeout and ideally in the session
header. The user should not have to read `AGENTS.md` to know what repo is being
mutated.

### 4. Protocol-First Responses Feel Like More Work

The Workbench docs are useful, but when the user is dogfooding, the first
answer should not be another protocol layer. It should start from the user
workflow:

```text
As a user, I want to resume this session and know whether it is safe to trust.
```

Then the agent can map that to Self-Awareness, Friction Tier, `.learning`, or
review rules. The order matters.

### 5. Accessibility/Automation Failure Needs A User-Facing Fallback

When Computer Use returns `Apple event error -1743`, the agent should say what
changed in confidence:

```text
I cannot inspect the live app window through accessibility. I can still use the
provided screenshot, repo state, and Superconductor MCP status.
```

That is better than silently switching evidence types.

## Cognitive Cache Extraction

This dogfood pass is exactly the kind of event Windburn's `.learning` layer
should preserve.

### Perception

The operator sees multiple overlapping truth surfaces: Superconductor project,
terminal session, worktree status, Git diff, and durable docs.

### Belief

Operator trust comes from a compact, current status card more than from the
existence of rich protocol docs.

### Failure Memory

Agent behavior can default to reviewer/protocol mode even when the user asks for
a lived product dogfood pass. Future runs should start dogfood from a user job,
not a repo checklist.

### Skill

For dogfooding:

```text
1. Name the user job.
2. Exercise or inspect the actual visible workflow.
3. Separate user friction from implementation defects.
4. Record evidence confidence and fallback paths.
5. End with one product primitive that would reduce operator effort.
```

### Parking

Add a Superconductor session status card that can be generated by combining:

- Superconductor worktree status;
- Git changed-file buckets;
- current agent objective/title;
- last command/evidence artifact;
- waiting-on-human flag;
- safe next action.

## Applicable Now: Goal Mode Card

Use this when a fuzzy dogfood thought appears in direct chat and the operator
wants it turned into immediate work without losing the user-seat perspective.

```text
/goal
GOAL_MODE: yes
STANDARD_PATH: yes

RAW_REQUIREMENT
Turn the current fuzzy dogfood observation into a concrete Superconductor
session trust improvement.

GOAL_LOCK:
objective: produce one current session status card and one reviewable product
  primitive that reduces operator state reconstruction.
owner: assigned agent in the current Superconductor worktree
non_goals: no daemon/Desktop/core runtime mutation; no credential access; no
  raw screenshot or transcript persistence; no broad redesign.
closeout_gates: user_job, visible_surfaces, changed_file_buckets,
  evidence_confidence, confidence_limits, safe_next_action, docs_report,
  git_status.
operator_call_conditions: ask only before live app mutation, destructive
  cleanup, publishing, or changing the operator's chosen repo anchor.

USER_JOB
Can I look at this session and know what the agent is doing, what changed, what
proof exists, and what I should do next?

STATUS_CARD
Goal:
Current action:
Last verified evidence:
Changed files:
Waiting on:
Safe next action:

VERDICT: PASS | FLAG | BLOCK
```

Default route:

- `FAST_PATH` if the output is only a status card or short note.
- `STANDARD_PATH` if docs, templates, or small scripts change.
- `HEAVY_PATH` only if the task touches live runtime mutation, credentials,
  publishing, daemon/Desktop/core behavior, or cross-agent automation.

## Dogfood Closeout Template

Use this compact closeout when acting like a user:

```text
USER_JOB:
VISIBLE_SURFACES:
WHAT_WORKED:
WHAT_FORCED_THE_USER_TO THINK:
EVIDENCE:
CONFIDENCE_LIMITS:
ONE_PRODUCT_PRIMITIVE:
VERDICT: PASS | FLAG | BLOCK
```

## Verdict

`FLAG`

The system is powerful and inspectable, but it still asks the operator to do too
much mental state reconstruction. The smallest valuable product improvement is
not another deep protocol. It is a visible session status card that answers:

```text
what is this agent doing, what changed, what proof exists, and what should I do
next?
```
